Testing for developers

Presentation by Debbie Roycroft @DebbieRoycroft

But I'm a developer- why do I need to know about testing?

  • The sooner bugs are found the cheaper they are to fix
  • Stop faults recurring
  • Make sure your code does what the customer wants
  • Boundaries between developer and QA are more blurred

So what testing do I need to do?

  • Many different kinds of testing
  • Not all will suit all projects/teams
  • Pick what works for you

Manual Testing

  • Code review
  • Running by hand and checking it does what it should
  • Time-consuming if you need to do it more than once
  • May not pick up knock-on effects in complex systems
  • Maybe we should look at automated testing...

Static Checking

  • Check code for 'simple' errors or code standards violations
  • Catch well-known gotchas
  • Helps keep code consistent and readable
  • Won't make sure your code works to spec

Static Checking Tools

  • IDE - IntelliJ, NetBeans, Eclipse
  • PMD
  • FindBugs
  • CheckStyle

Unit Testing

  • TDD or not TDD
  • Exercise small section of code
  • Each test is independent
  • Aim of each test should be clear
  • Treat unit test code like production code
  • Don't just exercise the 'happy path'
  • Cover edge cases
  • Cover exception conditions
  • Always use varied data
  • Take care with mocks/dummies
  • Test to the public interface
  • Don't be concerned with internals
  • Beware of singletons/global data
  • Design code with dependency injection
  • Code with small functions, single responsibility
  • Gives us confidence to refactor code
